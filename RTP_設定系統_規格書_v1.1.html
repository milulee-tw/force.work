01

功能概述

本文件描述遊戲管理後台中「RTP 動態設定」功能模組的完整設計規格，供產品、前端、後端及 QA 人員使用。所有設定異動均受版本控制與審計日誌保護。

功能目標
  • 允許具備 Admin 權限且在 RTP 白名單中的特定帳號，針對線路、幣值、遊戲、帳號群組等維度設定差異化 RTP 值
  • 採用 Priority 模型：條件越具體優先級越高，未命中時自動 fallback 至父層或全局預設值
  • 每條規則具備明確的生效起始與生效截止時間，截止後自動移入「已結束」列表
  • 帳號群組（Group）為獨立管理實體，可掛載多個特殊 Event（如強制進入 FG），Event 類型支援後續擴充
  • 支援條件維度後續擴充，不破壞現有規則
  • 提供規則衝突偵測、RTP 模擬器及完整操作審計日誌
適用範圍
項目說明
功能模組遊戲管理後台 → RTP 設定分頁
使用者Admin 角色，且被加入 RTP 設定白名單的帳號
影響範圍線上遊戲 RTP 機率，涉及合規、財務與玩家體驗
優先級High — RTP 設定直接影響出獎率與財務報表
02

RTP 優先序模型

系統採用「條件越具體，優先級越高」的 Cascading Priority 模型。玩家每次發起遊戲請求時，系統從最具體的規則向上查找，命中第一條有效規則即採用該 RTP 值。

優先序等級定義
優先級條件組合範例說明
P1group_id + game + line + currencygrp_vip + GemBonanza + pb10 + PHP最精確，針對特定群組/遊戲/線路/幣值
P2group_id + gamegrp_vip + GemBonanza針對特定群組在特定遊戲
P3group_idgrp_vip針對特定群組全部遊戲
P4line + currency + gamepb10 + PHP + GemBonanza針對特定線路/幣值/遊戲
P5line + currencypb10 + PHP針對特定線路/幣值
P6linepb10針對特定線路全部遊戲
P7gameGemBonanza針對特定遊戲全線路
P8（全局預設）—所有條件均未命中時的 fallback
⚠
當兩條規則優先級相同（例如同時設定 [pb10 + PHP] 和 [pb10 + GemBonanza]），且一個玩家請求同時命中兩者，系統需觸發衝突警告，不可靜默覆蓋。
Fallback 查找流程
P1 完整條件命中 → 採用此 RTP
↓ 未命中
P2 group_id + game 命中 → 採用此 RTP
↓ 未命中
P4 line + currency + game 命中 → 採用此 RTP
↓ … 依序往下
P8 全局預設 RTP（最終 fallback）
條件維度擴充規則
  • 維度越具體（針對個別玩家）優先級越高，新增維度需插入現有優先序之間
  • 新維度插入不得改變現有規則的相對優先序
  • 需同步更新本文件的優先序表及後端衝突偵測邏輯
  • 預留擴充維度：region（PH、TH、VN）等；Group 內的 Event 類型亦支援後續擴充

群組管理規格

群組（Group）是獨立管理的實體，與 RTP 設定同層。Admin 先在群組管理頁建立群組與成員，再於 RTP 設定中以 group_id 為條件建立規則。

審計日誌規格

所有對 RTP 規則的新增、修改、停用操作均需記錄審計日誌，日誌不可刪除，僅限 Admin 白名單帳號查閱。

日誌欄位定義
欄位名稱型別說明
log_idUUID日誌唯一識別碼
rule_idString操作對象的規則 ID
actionEnumCREATE / UPDATE / DEACTIVATE / ACTIVATE
operatorString執行操作的帳號
before_valueJSON操作前的規則快照（CREATE 時為 null）
after_valueJSON操作後的規則快照
ip_addressString操作者 IP
timestampDateTime操作時間（UTC+8）
查詢介面規格
  • 篩選欄位：操作者帳號、操作類型、rule_id、時間區間
  • 顯示欄位：timestamp、operator、action、rule_id；點擊可展開 before/after JSON diff
  • 不開放匯出功能，避免 RTP 策略洩漏

08

限制條件與合規要求

RTP 值邊界
  • 最小值：80.00%（0.80）
  • 最大值：99.00%（0.99）
  • 精度：小數點後 2 位
  • 超出範圍時，前後端均需阻擋並回傳錯誤
操作安全規則
  • 不開放批次操作，每次僅可編輯單條，降低誤操作風險
  • 不開放刪除，僅允許「停用」（is_active = false），保留歷史可追溯性
  • RTP 值 ≥ 98% 需額外輸入確認碼（CONFIRM）才可儲存
  • 同一帳號連續操作超過 10 條規則，需觸發系統通知（通知 Super Admin）
⚠
停用操作不可逆，已停用的規則移入「已結束」列表後不可重新啟用，需建立新規則才能恢復設定。
