Chapter 01
宏觀視角：為什麼這個計畫值得做

1.1 市場正在分化

SLOT 遊戲的競爭已從「誰的遊戲好玩」轉移到「誰能讓玩家每天回來」。八家競品的 CRM 功能盤點顯示，功能完整度高的廠商與低的廠商之間，玩家黏著度出現明顯分化。

最具說服力的數據來自 VP 奕樂——上線後活躍人流與押注量是同期無 CRM 系統廠商的 10 倍。每晚一個月上線，就是一個月的留存損失。

1.2 核心命題

讓玩家從「偶爾來玩」變成「每天有事要做」——靠的不是更多廣告，而是在遊戲內建立一套讓玩家主動回訪的目標感系統。

1.3 整體架構：三個支柱

支柱 1
行為目標系統
每日簽到、押注任務、排行榜、賽季競賽——讓玩家每天有具體目標可追求。
支柱 2
獎勵生態系統
第二貨幣、抽寶箱、道具卡、幸運輪——讓玩家對獎勵有累積感與期待感。
支柱 3
精準投放系統
Tag 標籤、動態 RTP 貢獻池、個人化通知——讓每個玩家收到針對他設計的體驗。
Chapter 02
五個核心概念與對應做法
概念一 第二貨幣：從被動領獎到主動管理資產

核心思維：直接給現金或道具卡，玩家是被動接受獎勵。第二貨幣（幸運幣）把這件事反轉：玩家變成主動管理自己的資產，差別不在金額，而在玩家有沒有動力採取行動。

統一出口：所有 CRM 場景（簽到、任務、排行榜、分享）都發同一種幣，玩家只管理一個餘額，認知負擔最低。

累積機制：「今天開銅箱」或「再等三天開金箱」的選擇，在玩家腦中製造未來回訪的計畫。

每日獲取上限控制：依玩家當日押注量或 VIP 階級設定每日可獲幸運幣上限，防止異常漏洞行為，同時確保幣的稀缺感。VIP 越高，上限越高，形成差異化激勵。

幣水庫平衡：必須持續監控「發行量 vs 消耗量」的動態平衡。發行量持續大於消耗量，幣會快速通膨、激勵失效；消耗量過快，玩家感受不到累積感。建立水庫觀測看板，從 S2 上線起即開始追蹤。

概念二 隨機消耗機制：驚喜感是留存的核心燃料

核心思維：SLOT 玩家已習慣「平時小贏、偶爾大爆」的心理節奏。固定兌換的結果已知，沒有驚喜，本質上和直接給卡沒有差別。隨機性本身才是驅動玩家累積幣的動力。

Phase 1 · 抽寶箱（輕量快速上線）：三個等級（銅／銀／金箱）對應不同幣成本，稀有率遞增。保底機制：連續 N 次未中稀有，第 N+1 次保證中。開發量小（3–5 天），心理機制與幸運輪幾乎相同。

Phase 4 · 幸運輪（儀式感升級版）：玩家已習慣抽箱六個月後推出，此時推出效果最大。幸運輪不是第一個消耗場景，而是「終局升級體驗」。

概念三 Tag 標籤系統：CRM 的精準基礎設施

核心思維：沒有 Tag 的 CRM 是廣播。Tag 是瞄準鏡——讓對的玩家在對的時機收到對的內容。作為遊戲商，我們無法對離線玩家發送站外推播，Tag 觸發的動作必須設計為「玩家回來那一刻自動生效」。

S1 最小可行版（3 個核心 Tag）：新玩家（≤7天）、斷簽風險（超過 2 天未簽到）、高押量（鯨魚）。三個 Tag 衍生的 CRM 動作已足夠驗證效果。

斷簽風險 Tag 的正確觸發邏輯：玩家重新登入的當下，系統偵測斷簽天數，自動顯示幸運幣補簽提示（Phase 1），或優先消耗補簽卡補回（Phase 3）。斷簽 1 天補 50 幣，斷簽 2 天補 80 幣，斷簽 3 天以上觸發回歸禮包（幣 + 低門檻任務）。所有動作均在玩家主動登入後才觸發，無需站外推播能力。

兩個層次分開設計：「運營行為標籤」（系統自動打標，玩家看不到）從 S1 開始建立；「玩家可見徽章」（成就感展示）Phase 3 再做，避免範圍爆炸。

概念四 動態 RTP 貢獻池：把遊戲回報轉化為活動獎勵

核心思維：每個活動從遊戲基礎 RTP 中「撥出」一個比例進入獎池。玩家參與的活動越多，即時 RTP 越低——但那部分 RTP 並未消失，而是進入任務獎勵、Jackpot 彩金等活動的獎池，等待兌現。

注單 RTP 紀錄（v3 新增）：每一筆押注注單必須記錄當下實際使用的 RTP（即扣除所有活動撥出後的即時 RTP）。目的是確保在玩家同時接多個活動時，有可追溯的 RTP 歷史，避免超出下限保護的爭議，並提供運營後台的活動效益分析依據。

資格門檻：Jackpot 等活動設最低押注門檻，押注不達標的玩家不被扣除 Jackpot 撥出部分，維持較高即時 RTP。

硬下限保護：各活動撥出比例加總不得使任何玩家即時 RTP 低於設定下限（建議依 VIP 等級設定差異化下限）。超過時，後端自動停止計入新活動撥出。此下限寫死於 Server 端，不可人工突破。

留存問題不是一次性解決的，不同階段的玩家面對不同的流失原因。四個 Phase 分別針對習慣建立、目標感、資產依附感、長期成長路徑，按順序推進。

Chapter 06
執行原則與注意事項
1

幣上線與消耗場景同一天

S2（幣系統）和 S3（抽寶箱 + 輕量道具欄）必須同一天上線，消除空窗期，確保玩家第一天賺到幣就能花出去，且花完有地方看。

2

RTP 下限由 Server 鎖定

各活動撥出比例加總不得使玩家即時 RTP 低於設定下限，由後端程式強制執行，無法人工突破。可依 VIP 等級設定差異化下限。

3

每季根據數據動態調整

本計畫是框架，不是固定劇本。每季末 KPI 回顧後，動態調整 Phase 優先序與各活動 RTP 撥出比例是否合理，幣水庫水位納入調整依據。

動態 RTP 設計的常見錯誤

錯誤後果正確做法
多個活動撥出過大，未設下限玩家即時 RTP 過低，體驗變差直接流失設定 RTP 下限，後端鎖定，超過即停止計入
Jackpot 對所有人撥出，不分押注資格小押注玩家被扣 RTP 但無緣彩金，不公平Jackpot 撥出僅適用達門檻玩家，系統自動判斷
撥出比例與獎金預算不匹配撥出 1% 但實際發獎遠低於此，平台財務有損撥出比例 = 預計發獎總額 ÷ 預計總押注量
沒有注單 RTP 紀錄無法追蹤玩家實際體驗到的即時 RTP，活動效益無從分析每筆押注注單記錄當下實際 RTP，供後台查詢
第二貨幣無每日發放上限異常漏洞行為、幣快速通膨、激勵效果失效依押注量或 VIP 設定每日獲取上限，並監控水庫水位
斷簽召回設計為站外推播遊戲商無法觸達離線玩家，機制形同虛設所有召回動作設計為「玩家登入當下自動觸發」：幣補簽提示（P1）→ 補簽卡（P3）

遊戲商 CRM 的核心限制：無站外觸達能力

作為遊戲商（非平台方），我們無法對已離線的玩家發送任何站外推播或通知。所有 CRM 動作必須設計為「玩家主動回來的那一刻才觸發」，而非主動追出去。這是本計畫所有召回設計的根本前提。

Chapter 07
人力配置需求（四季估算）

以下估算基於每季 40 個工作天、單位為「人天（person-days）」。含 v3 新增的輕量道具欄（S3）、道具欄強化（S8）工作量。

Q1｜Phase 1：回訪基礎（S1–S4）

本季核心難度在後端：Tag引擎（含斷簽偵測與幣補簽觸發）、RTP撥出引擎（含注單RTP紀錄）、第二貨幣帳本（含每日上限）、抽寶箱機率引擎，同時還需交付輕量道具欄，是全年技術複雜度最高的一季。

職能估算人天主要工作內容
企劃（PM+企劃）80天四個系統需求文件、UI流程規劃、KPI定義、Sprint Review、幣發放/消耗規則設計
平面美術28天簽到UI、幸運幣圖示+錢包視覺、三階寶箱設計、開箱動效素材、輕量道具欄 UI、Popup模板
Web 前端26天簽到頁面、幣錢包頁面（含輕量道具欄整合）、Inbox列表/詳情頁、Popup觸發前端邏輯
Unity18天旋轉卡道具化（in-game卡牌系統整合）、開箱動畫接入、遊戲內幣餘額顯示
後端50天Tag引擎（自動打標 + 斷簽偵測 + 登入觸發幣補簽）、RTP撥出計算引擎（即時扣除+下限鎖定+注單RTP紀錄）、第二貨幣帳本（含每日上限）、抽寶箱機率引擎+保底機制
數據分析14天基礎留存/簽到監控看板、幣水庫觀測表初版（發行量 vs 消耗量）
⚠ Q1 後端風險

後端50天對單人幾乎是滿載（40天/季）。強烈建議 Q1 配置後端 1.5 人力，或讓後端提前一個 Sprint 預先設計 RTP 引擎的資料模型與 API 規格，再進入正式開發。
